OPENDOG-多项目文件监控系统（WSL+Rust+MCP）详细开发方案

原生需求：
我正在用AI开发一个项目，我想给这个项目增加一个常驻功能，它先对项目内的所有文件进行一次快照，读取基本信息，然后监控当前AI调用/使用的所有文件的次数/时长/修改，要求尽量不阻断原命令的操作，只观察/监控该命令涉及的文件，以此来生成统计信息，包括快照列表中的文件引用的次数/时长，由于当前项目中的文件太多，我不知道哪些文件是垃圾/过时/有用，所以想用这个方法，了解项目中文件的使用情况，监控一段时间后，对于那些从未被引用的文件，再另外进行集中分析和处理。你是软件开发大师，对此有什么意见，我该如何实现，我使用的是codex + gpt-5.4,  claude code +glm-5.1

一、方案核心定位与目标
1.1 核心定位
基于WSL环境，采用Rust语言开发，以MCP（Model Control Protocol）服务为核心形态，构建一款高性能、无侵入、多项目独立隔离的文件监控系统，专门适配AI项目（Codex + GPT-5.4 + Claude Code + GLM-5.1）的文件使用监控需求。
1.2 核心目标
- 多项目隔离：多个AI项目分开管理，数据库、配置、监控任务完全独立，互不干扰。
- 无侵入监控：不阻断原AI命令操作，仅透明观测AI调用涉及的文件，记录访问次数、使用时长、修改行为。
- 高性能常驻：后台低功耗运行（CPU占用<1%，内存占用<10MB），支持海量文件秒级快照，7×24小时稳定常驻。
- AI友好：提供MCP标准化接口，支持AI工具直接调用，实现监控任务的自动化管理与统计查询。
- 可落地性：基于WSL生态，部署简单、维护便捷，适配现有AI开发环境，无需改造原有项目结构。

三、整体架构设计（分层+多项目隔离）
3.1 架构分层（从下至上，低耦合）
（1）基础层：运行环境与依赖
基于WSL系统，搭建Rust编译环境、Systemd服务管理环境，引入核心依赖库（文件监控、数据库、MCP服务、CLI等），为上层功能提供基础支撑，确保系统可编译、可运行、可常驻。
（2）核心层：多项目管理与监控引擎
系统核心，负责多项目的隔离管理、文件快照生成、实时文件监控，是整个系统的核心逻辑载体，分为两个核心模块：
- 多项目管理模块：维护所有监控项目的信息，为每个项目分配独立ID、独立根目录、独立数据库文件、独立配置，实现项目间的完全隔离；提供项目的创建、删除、查询、状态管理等功能。
- 监控与快照模块：为每个项目独立启动监控线程，通过inotify系统级事件监听，透明捕捉AI工具访问文件的行为（打开、读取、修改、关闭），不阻断原命令执行；生成项目文件快照（记录文件路径、大小、修改时间、文件类型等基线信息），并实时更新文件使用统计数据（访问次数、使用时长、修改次数、最后访问时间等）。
（3）服务层：MCP服务与CLI接口
对外提供操作入口，分为两个接口类型，适配不同使用场景：
- MCP服务接口：标准化API，支持AI工具（Codex/Claude/GPT/GLM）直接调用，实现监控任务的自动化管理（创建项目、启动/停止监控、生成快照、查询统计数据等）。
- CLI接口：命令行操作入口，支持手动管理项目（创建项目、启动监控、查看未使用文件等），适配非AI调用场景，方便手动维护。
（4）存储层：多项目独立数据库
采用SQLite多数据库文件模式，每个项目对应一个独立的.db文件，存储该项目的两个核心数据表：
- 文件快照表：存储项目文件的基线信息（路径、大小、修改时间、文件类型、首次扫描时间），作为后续监控对比的基准。
- 文件统计表：存储文件的使用数据（访问次数、总使用时长、修改次数、最后访问时间、是否被访问过），用于后续统计分析与垃圾文件识别。
3.2 多项目隔离设计（核心重点）
确保多个项目互不干扰，核心实现方式如下：
1. 命名空间隔离：每个项目分配唯一项目ID，所有与项目相关的资源（数据库文件、配置、监控线程）均以项目ID为前缀命名，避免资源冲突。
2. 数据隔离：每个项目对应独立的SQLite数据库文件，存储该项目的快照与统计数据，不同项目的数据库文件互不关联，即使一个项目的数据库出现问题，不影响其他项目正常运行。
3. 任务隔离：每个项目启动独立的监控线程，线程间互不干扰，可单独启动、停止某个项目的监控，不影响其他项目的监控任务。
4. 配置隔离：每个项目拥有独立的配置（项目根目录、忽略目录、监控参数等），存储在独立的配置文件中，可针对不同项目的需求单独调整配置。
四、核心功能设计（分模块详细说明）
4.1 多项目管理功能
核心作用：实现多个AI项目的独立管理，支持项目的全生命周期操作，适配多项目并行监控需求。
- 项目创建：输入项目ID、项目根目录，系统自动创建该项目的独立数据库文件、配置文件，初始化数据库表结构，完成项目的初始化。
- 项目查询：支持查询所有已创建的项目，查看每个项目的状态（监控中/已停止）、根目录、数据库路径等信息。
- 项目启停：可单独启动、停止某个项目的监控任务，启动时自动加载该项目的配置与数据库，停止时释放监控资源，不影响其他项目。
- 项目删除：删除指定项目，同时删除该项目的数据库文件、配置文件，彻底清理该项目的所有资源，不残留垃圾文件。
4.2 文件快照功能
核心作用：生成项目文件的基线清单，作为后续监控、统计的基准，支持海量文件高效扫描。
- 全量快照：递归扫描项目根目录下的所有文件，自动过滤无用目录（node_modules、.git、dist、__pycache__等），记录文件的核心信息（路径、大小、修改时间、文件类型、首次扫描时间），存入该项目的快照表。
- 快照更新：支持手动触发快照更新，更新项目中新增、删除、修改的文件信息，确保基线清单与项目实际文件一致。
- 快照校验：扫描过程中自动跳过无法访问、权限不足的文件，避免扫描异常中断，确保快照的完整性。
4.3 实时监控功能（无侵入核心）
核心作用：透明监控AI工具访问项目文件的行为，不阻断原命令执行，精准记录文件使用数据。
- 监控范围：仅监控当前项目根目录下的文件，且仅捕捉AI工具（Codex、GPT-5.4、Claude Code、GLM-5.1）相关进程访问的文件，过滤其他无关程序的文件操作。
- 监控维度：
        
  - 访问次数：记录文件被AI工具打开/读取的次数，每次打开即计数一次。
  - 使用时长：记录文件从打开到关闭的耗时，累计计算总使用时长。
  - 修改行为：记录文件被AI工具修改的次数，每次修改即计数一次，同步更新文件修改时间。
  - 访问时间：记录文件最后一次被AI工具访问的时间，用于判断文件的活跃状态。
- 无侵入实现：采用系统级inotify事件监听，仅被动接收文件操作事件，不拦截、不修改原AI命令的执行流程，不影响AI工具的正常使用，实现完全透明监控。
- 异常处理：监控过程中自动忽略临时文件、缓存文件的操作，避免统计数据失真；出现文件删除、路径变更时，自动更新快照与统计数据，确保监控连续性。
4.4 统计分析功能
核心作用：基于监控数据，生成文件使用报告，精准识别垃圾文件、核心文件，为项目文件清理提供数据支撑。
- 核心文件识别：统计高频访问、长使用时长的文件，标记为核心文件（AI开发常用文件），方便重点保护。
- 垃圾/过时候选识别：统计从未被AI工具访问的文件（快照中存在、统计表中无访问记录），标记为垃圾/过时候选文件，供后续集中分析处理。
- 统计报告生成：支持按项目生成统计报告，包含项目文件总数、核心文件数量、未使用文件数量、各文件的访问次数/时长等信息，直观展示文件使用情况。
- 数据导出：支持将统计数据导出为通用格式（如CSV），方便后续手动分析或导入其他工具处理。
4.5 MCP服务接口功能
核心作用：提供标准化接口，支持AI工具直接调用，实现监控任务的自动化管理，适配AI开发流程。
核心接口清单（标准化设计，AI可直接调用）：
- create_project：创建新项目，参数为项目ID、项目根目录，返回项目创建结果。
- take_snapshot：为指定项目生成/更新快照，参数为项目ID，返回快照完成状态与文件总数。
- start_monitor：启动指定项目的监控任务，参数为项目ID，返回监控启动状态。
- stop_monitor：停止指定项目的监控任务，参数为项目ID，返回监控停止状态。
- get_stats：获取指定项目的文件使用统计数据，参数为项目ID，返回核心文件、未使用文件等统计信息。
- get_unused_files：获取指定项目的从未使用文件列表，参数为项目ID，返回文件路径列表。
- list_projects：列出所有已创建的项目，返回项目ID、状态、根目录等信息。
- delete_project：删除指定项目，参数为项目ID，返回删除结果。
4.6 CLI命令行功能
核心作用：提供本地手动操作入口，方便非AI调用场景下的项目管理与监控操作，适配手动维护需求。
核心命令清单（简洁易用，支持参数提示）：
- file-monitor create --id 项目ID --path 项目根目录：创建新项目。
- file-monitor snapshot --id 项目ID：为指定项目生成/更新快照。
- file-monitor start --id 项目ID：启动指定项目的监控。
- file-monitor stop --id 项目ID：停止指定项目的监控。
- file-monitor stats --id 项目ID：查看指定项目的统计报告。
- file-monitor unused --id 项目ID：查看指定项目的未使用文件列表。
- file-monitor list：列出所有项目的状态信息。
- file-monitor delete --id 项目ID：删除指定项目。
五、部署与运维方案（WSL环境，简单易操作）
5.1 部署步骤（分3步，一键可完成）
1. 环境准备：在WSL中安装Rust编译环境（cargo）、SQLite依赖、Systemd服务管理工具，确保WSL可正常运行Linux命令。
2. 编译打包：将Rust项目代码编译为可执行文件（release版本，优化性能），生成相关配置文件模板、数据库存储目录。
3. 服务注册：将可执行文件注册为Systemd服务，配置开机自启、故障自动重启，确保服务长期稳定常驻；配置服务日志输出，方便后续排查问题。
5.2 运维管理（低成本，无需复杂操作）
- 服务监控：通过Systemd命令查看服务运行状态（systemctl status 服务名），查看日志（journalctl -u 服务名），排查运行异常。
- 数据备份：定期备份各项目的数据库文件（.db），避免数据丢失；可设置自动备份脚本，定时将数据库文件备份至指定目录。
- 版本更新：后续如需新增功能，重新编译Rust代码，替换原有可执行文件，重启Systemd服务即可，不影响现有项目的监控数据。
- 异常处理：若某个项目的监控出现异常，可单独停止该项目的监控，排查配置或文件路径问题，重启该项目监控即可，不影响其他项目。
六、关键优化与注意事项
6.1 关键优化（提升系统性能与易用性）
- 监控性能优化：采用异步监听机制，避免单线程阻塞；仅监控AI相关进程的文件操作，过滤无关操作，降低系统资源占用。
- 快照性能优化：递归扫描时采用多线程并行扫描，支持海量文件秒级快照；自动过滤无用目录，减少扫描数据量。
- AI兼容性优化：适配Codex、GPT-5.4、Claude Code、GLM-5.1等AI工具的进程特征，精准捕捉AI的文件操作，避免统计失真。
- 资源占用优化：Rust编译时开启release优化，降低后台常驻的CPU、内存占用；数据库采用轻量级SQLite，无需额外占用系统资源。
6.2 注意事项（避免踩坑，确保系统稳定）
- 项目路径配置：确保项目根目录路径正确，且WSL具有该路径的读写权限，否则无法正常扫描快照、监控文件。
- 进程过滤：准确配置AI工具的进程名称，避免遗漏或误判，确保仅统计AI工具的文件操作。
- 数据隔离：禁止手动修改数据库文件，避免破坏数据完整性；删除项目时，确保彻底删除相关数据库与配置文件，避免残留。
- WSL稳定性：确保WSL正常运行，避免WSL重启导致服务中断；可配置Systemd服务自动重启，减少人工干预。
- 文件清理：未使用文件标记为垃圾/过时候选后，建议先备份再清理，避免误删核心文件（即使未被访问，也可能存在潜在用途）。
